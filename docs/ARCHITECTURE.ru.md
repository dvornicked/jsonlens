# JSONLens — Архитектура

Подробный разбор того, как JSONLens плавно рендерит **десятки мегабайт** JSON, и
_почему_ каждая часть сделана именно так. Если читать только один раздел — читайте
[Главная идея](#главная-идея).

> Английская версия: [ARCHITECTURE.md](./ARCHITECTURE.md).

---

## Проблема

Наивный JSON-вьювер делает две вещи, которые разваливаются на масштабе:

1. **`JSON.parse` в главном потоке.** Парсинг 30 МБ занимает ~150–400 мс и даёт
   всплеск памяти: каждый объект/массив/строка становится JS-значением в куче с
   оверхедом заголовка на объект, поэтому 30 МБ _текста_ превращаются в 150–300 МБ
   _кучи_. В UI-потоке это вешает вкладку.
2. **Один DOM-узел на каждый JSON-узел.** Документ в 30 МБ — это миллионы узлов.
   Создавать `<div>` на каждый — миллионы layout-боксов; браузер умрёт задолго до
   того, как пользователь начнёт скроллить.

Всё, что ниже, существует, чтобы обойти эти две ловушки.

---

## Главная идея

> **Распарсить один раз, вне главного потока, в плоскую индексированную модель.
> Рендерить только строки во вьюпорте. Сделать collapse/expand/скролл за
> `O(log n)`, никогда не `O(n)`.**

Конкретно:

- Данные живут в **Web Worker** как **preorder struct-of-arrays** модель на
  типизированных массивах — а не дерево JS-объектов.
- UI — тонкий **виртуализированный список**: просит у воркера «видимые строки в
  `[start, start+count)`» и рендерит ~50 DOM-узлов независимо от размера документа.
- **Дерево Фенвика** (binary indexed tree) над битом «виден ли узел» позволяет за
  `O(log n)` отображать _индекс строки → узел_ и _узел → индекс строки_, а также
  переключать видимость целого поддерева за время, пропорциональное числу строк,
  которые реально появились/исчезли.

UI никогда не видит материализованное дерево объектов — только **строки по индексу**
и **непрозрачные id узлов**. Именно эта граница позволяет позже подменить движок на
Rust/WASM, не трогая вьювер.

## Обзор конвейера

```
 сырая JSON-страница
      │
      ▼
┌───────────────────┐   детектит JSON-документ, заменяет страницу,
│  content script   │   монтирует Preact-вьювер (или восстанавливает при битом JSON)
│  content/content  │
└─────────┬─────────┘
          │ текст документа
          ▼
┌───────────────────┐   Preact-UI: тулбар, поиск, виртуализированное дерево.
│   viewer (App)    │   Общается с воркером через EngineClient (промисы поверх
│  viewer/*.tsx     │   postMessage), рендерит только строки вьюпорта.
└─────────┬─────────┘
          │ сообщения (parse / getRows / toggle / search / reveal / copy / serialize)
          ▼
┌───────────────────┐   крутит Engine вне главного потока.
│   Web Worker      │
│  worker/*.ts      │
└─────────┬─────────┘
          │ интерфейс Engine
          ▼
┌───────────────────┐   JSON.parse → preorder struct-of-arrays модель;
│   TsEngine        │   видимость на дереве Фенвика. (Заменяемо на WasmEngine.)
│  engine/*.ts      │
└───────────────────┘
```

## 1. Content script — детект и захват страницы

`src/content/content.tsx` запускается на `document_end` на каждой странице
(манифест матчит `<all_urls>`), но сразу выходит, если страница не JSON:

- Он читает `body.textContent` только когда страница **в принципе могла бы быть**
  JSON — то есть content-type из семейства JSON (`application/json`, `text/json`
  или любой `+json`-суффикс) **или** body — единственный `<pre>` (форма, в которой
  браузеры рендерят сырой текст). Это избавляет от сериализации всего DOM каждой
  обычной HTML-страницы только ради того, чтобы его выбросить.
- Для не-JSON content-type дополнительно требуется _структурность_ текста
  (`{…}` / `[…]`), чтобы не захватывать произвольный `text/plain`.

**Восстановление при ошибке.** Детект — дешёвая проверка формы, не полный парс.
Поэтому `mount()` **сохраняет исходные узлы документа**, очищает страницу,
рендерит вьювер и передаёт колбэк `onParseError`. Если воркер позже сообщит, что
текст не валидный JSON, вьювер дёргает колбэк и исходная страница восстанавливается
— никакого невосстановимого blank из-за `Content-Type: application/json`, который
на деле является HTML-страницей ошибки.

## 2. Граница воркера

Движок крутится в `Worker`, чтобы парсинг и индексация никогда не касались
UI-потока.

`EngineClient` (`src/viewer/engineClient.ts`) оборачивает голый message-port в
промис-API: каждый запрос получает correlation-id, и нужный ответ резолвит свой
промис. Исключения на стороне воркера приходят как `error`-ответ, который
**реджектит** ожидающий промис, а обработчик `worker.onerror` реджектит всё
в полёте — так что брошенное в движке исключение не оставит UI висеть на спиннере.

### Почему Blob-воркер (неочевидная часть)

Origin документа в content-script — это _страница-хост_ (например,
`https://site.com`). Скрипт `Worker` должен быть same-origin; даже
web-accessible `chrome-extension://…` URL **блокируется** при создании воркера из
content-script. И `new Worker(new URL('…', import.meta.url))` тоже не работает —
content-script не является ES-модулем, поэтому `import.meta` бросает исключение.

Бандл воркера полностью **самодостаточен** (ноль импортов), поэтому клиент
**фетчит его код** через `chrome.runtime.getURL(...)` и запускает из same-origin
**Blob URL**:

```ts
const code = await (await fetch(chrome.runtime.getURL(workerUrl))).text();
const worker = new Worker(URL.createObjectURL(new Blob([code], { type: 'text/javascript' })));
```

Это поймал именно Playwright E2E-тест — без этого расширение молча не монтировалось.

## 3. Плоская модель (сердце)

`TsEngine.parse` (`src/engine/ts-engine.ts`) один раз выполняет `JSON.parse`,
затем **однократно** обходит дерево значений в параллельные типизированные массивы
— struct-of-arrays раскладку, индексированную по id узла (его **preorder-позиции**):

| массив        | тип           | смысл                                         |
| ------------- | ------------- | --------------------------------------------- |
| `kind`        | `Uint8Array`  | Object/Array/String/Number/Bool/Null          |
| `depth`       | `Uint32Array` | уровень вложенности                           |
| `parent`      | `Int32Array`  | id родителя (`-1` для корня)                  |
| `childCount`  | `Uint32Array` | число детей (0 у примитивов)                  |
| `subtreeSize` | `Uint32Array` | узлов в поддереве, **включая себя**           |
| `collapsed`   | `Uint8Array`  | свёрнут ли контейнер                          |
| `visible`     | `Uint8Array`  | рендерится ли узел сейчас                     |
| `keys`        | `string[]`    | ключ члена объекта / метка индекса массива    |
| `keyIsIndex`  | `Uint8Array`  | отличает индекс массива от строкового ключа   |
| `preview`     | `string[]`    | готовый к показу текст примитива              |
| `valueRef`    | `unknown[]`   | ссылка на живое JS-значение (для копирования) |

**Почему struct-of-arrays + типизированные массивы?** Числа лежат в непрерывных
буферах типизированных массивов вместо миллионов боксированных JS-объектов — это
срезает оверхед на узел и давление на GC, ровно то, что убивает наивный подход.
И это та раскладка, которую хочет WASM-движок (те же столбцы ложатся в линейную
память).

### Preorder = непрерывные поддеревья

Самый полезный инвариант: поскольку узлы нумеруются в **preorder**, всё поддерево
узла занимает непрерывный диапазон id `[i, i + subtreeSize[i])`. Отсюда:

- «Свернуть узел `i`» = «спрятать диапазон `(i, i + subtreeSize[i])`».
- «Пропустить свёрнутое поддерево» = «прыгнуть с `i` на `i + subtreeSize[i]`».

Никаких указателей на детей и рекурсии для обхода или пропуска поддеревьев.

### Итеративная сборка с ограниченным стеком

И проход подсчёта узлов, и проход уплощения используют **явный стек-курсор**,
ограниченный _глубиной_ дерева (а не шириной), поэтому массив на 10M элементов не
переполняет стек вызовов. `subtreeSize[i]` заполняется, когда курсор контейнера
исчерпан: `subtreeSize[i] = nextId - i`.

## 4. Дерево Фенвика — O(log n) на всё

Вьюпорту постоянно нужны два отображения:

- **строка → узел**: «какой узел на видимой строке 12 034?» (при скролле)
- **узел → строка**: «на какой строке этот узел?» (после reveal / для подсветки)

и collapse/expand должны их обновлять. Плоский «список видимых id» делал бы каждый
toggle пересборкой за `O(n)`. Вместо этого держим
[дерево Фенвика](../src/shared/fenwick.ts) над массивом бит `visible`:

- `prefix(i)` — число видимых узлов в `[0, i]` → **узел → строка** это
  `prefix(id) - 1`.
- `findKth(k)` — id узла на видимой строке `k` (спуск lower-bound по дереву) →
  **строка → узел**.
- `update(i, ±1)` — переключить видимость одного узла.

Все три — `O(log n)`. `initFrom` строит дерево за `O(n)` (используется на
операциях над всем деревом — expand/collapse-all).

### Стоимость collapse / expand

`setCollapsed(i, collapse)` обходит поддерево `i`, но **пропускает вложенные
свёрнутые поддеревья** (прыгая по `subtreeSize`), затрагивая только строки, чья
видимость реально меняется, с одним `O(log n)` обновлением Фенвика на каждую. Так
сворачивание узла, прячущего _k_ видимых строк, стоит `O(k · log n)` —
пропорционально изменению, а не размеру документа. (Будущий range-update Фенвик
сделал бы это плоским `O(log n)`; см. `PLANS.md`.)

## 5. Виртуализация

`VirtualTree.tsx` рендерит спейсер высотой `visibleCount × ROW_H`, чтобы скроллбар
отражал весь документ, но материализует только строки в зоне видимости:

1. На скролле (или изменении модели) вычисляет первую видимую строку из `scrollTop`
   плюс overscan-запас.
2. Запрашивает у воркера это окно через `getRows(start, count)`.
3. Абсолютно позиционирует каждую вернувшуюся строку на `rowIndex × ROW_H`.

Монотонно растущий **request-token** отбрасывает устаревшие ответы `getRows`,
пришедшие не по порядку при быстром скролле. `count` зажимается до `≥ 0`, чтобы
модель, сжавшаяся под позицией скролла (например, collapse-all при прокрутке вниз),
не запросила отрицательное окно.

Фиксированная высота строки (`ROW_H`) — то, что держит арифметику виртуализации за
`O(1)`, без помодульного измерения строк.

## 6. Фичи, отображённые на модель

- **Поиск** — `engine.search(query, valuesToo)` сканирует ключи (и превью
  примитивов), возвращает упорядоченные id-совпадения; UI подсвечивает их, а
  `reveal()` разворачивает свёрнутых предков, чтобы вывести активное совпадение на
  экран.
- **Копирование** — `valueOf(id)` возвращает живое JS-значение (сериализуется для
  копии); `pathOf(id, flavor)` идёт по `parent`/`keys`, строя путь в JS (`a.b[0]`)
  или JSONPath (`$['a']['b'][0]`), с корректным экранированием ключей, где нужно.
- **Экспорт** — `serialize(pretty)` пере-сериализует весь документ **в воркере**
  (так что сериализация многомегабайтного JSON не блокирует UI) для скачивания в
  pretty/minified и для «копировать всё»; вариант «original» отдаёт исходные байты
  напрямую из `text` вьюера, байт-в-байт.
- **Collapse/expand all и to-depth** — видимость всего дерева пересчитывается за
  `O(n)`, дерево Фенвика пересобирается через `initFrom`.

## 7. Граница движка (и Rust/WASM)

`Engine` (`src/engine/types.ts`) намеренно узок и дружелюбен к zero-copy:
`getRows`, `toggle`, `search`, `reveal`, `pathOf`, `serialize`, … — **только
индексы, id и строки целого документа**, никогда деревья объектов. `TsEngine` — одна реализация; `WasmEngine` (Rust +
парсинг `serde_json`/`simd-json` прямо в линейную память) может реализовать тот же
интерфейс и подмениться в воркере **без изменений вьювера**. В этом весь смысл
границы, и поэтому бенч-харнес ([`bench/`](../bench)) меряет движок в изоляции —
чтобы выигрыши (или проигрыши) WASM-порта были видны напрямую.

## Сводка сложности

| операция                | стоимость                     |
| ----------------------- | ----------------------------- |
| парс + сборка модели    | `O(n)` (вне потока)           |
| `getRows(start, count)` | `O(count · log n)`            |
| `rowOf` / `findKth`     | `O(log n)`                    |
| toggle одного узла      | `O(изменённых строк · log n)` |
| collapse/expand all     | `O(n)`                        |
| поиск                   | `O(n)` (на запрос)            |

Где `n` — общее число узлов.

## Известные компромиссы

См. **Known limitations** в [`PLANS.md`](../PLANS.md): поиск по значениям на
основе превью (обрезка в 200 символов, числа переформатированы), копия/превью
отражают распарсенное значение, а не исходные байты (дубли ключей схлопнуты,
целочисленно-подобные ключи переупорядочены), и страницы со строгим CSP, способные
заблокировать Blob-воркер. У каждого есть запланированный фикс; ни один не ломает
ядро модели.
